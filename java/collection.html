collection

Created Sunday 13 March 2022

Instance of the two interfaces java.util.Collection and java.util.Map form collection framework.
The two interfaces are not related from a class hierarchical point of view, i.e. Map does not extend Collection, nor does Collection extend Map. That said, both interfaces are part of the Java Collection Framework.

Note : The Collection is an interface whereas Collections is a utility class in Java. Java collection class contains static methods that operate on collection frameworks.

As easy way to remember collections is IC-SQL-M (Iterable - Collection - Set Queue list - Map)



The Java Iterable interface represents a collection of objects which is iterable - meaning which can be iterated. This means that a class that implements the Java Iterable interface can have its elements iterated.
The Collection interface extends Iterable and hence all child classes of Collection also implement Iterable.

The Java Collections Framework most commonly used implementation :

  1. Set interface -> HashSet
  2. Queue+List interface -> LinkedList
  3. List interface -> ArrayList
  4. Deque interface -> ArrayDeque
  5. Map interface -> HashMap

Note - The name deque is short for "double ended queue" and is usually pronounced "deck".This interface defines methods to access the elements at both ends of the deque.

Set is a collection that cannot contain duplicate elements
The Java platform contains three general-purpose Set implementations: HashSet (unique), TreeSet(sorted), and LinkedHashSet(order).

HashSet internally uses HashMap {HM(K,V)} as a backing data structure with key as the supplied value hashcode [ HashSet put(value] ] and value as dummy Object class type.

HashSet put(value) --> internally --> Generate HashCode for value hashcode(value) = hc --> uses hashmap HashMap put(hc,dummyValue), dummyValue generated by internal implementation.
HashSet offers constant time performance for the basic operations like add, remove, contains, and size, but no get() method as there is no key-value pair [internal implementation uses hashmap/hashtable but actual value (user supplied) acts as a key for dummy values].

Difference between HashSet and ArrayList ?
First, HashSet, unlike ArrayList, is a Set: It cannot contain duplicates while ArrayList can - so they are built for different purposes. It also does not guarantee insertion order - again, unlike a list.

Map : HashMap and Hashtable(obsolete) store key and value pairs in a hash table. When using a Hashtable or HashMap, we specify an object that is used as a key and the value that you want to be linked to that key. The key is then hashed, and the resulting hash code is used as the index at which the value is stored within the table. HashMap is much faster and uses less memory than Hashtable as the former is unsynchronized . Unsynchronized objects are often much better in performance in comparison to synchronized objects like Hashtable in a single threaded environment.
Note : HashMap must not have duplicate keys, but it may have duplicate values.

A List is an ordered collection of elements. The ArrayList class is a resizable array, which can be found in the java.util package. The difference between a built-in array and an ArrayList in Java, is that the size of an array cannot be modified.

General Rule : "if it fits into an ArrayList (insertion ORDER) , use ArrayList, otherwise you probably need a HashMap (UNORDER)."

Difference between HashMap and HashTable ?
There are several differences between HashMap and Hashtable in Java:
Hashtable is synchronized, whereas HashMap is not. This makes HashMap better for non-threaded applications, as unsynchronized Objects typically perform better than synchronized ones.
Hashtable does not allow null keys or values. HashMap allows one null key and any number of null values.
One of HashMap's subclasses is LinkedHashMap, so in the event that you'd want predictable iteration order (which is insertion order by default), you could easily swap out the HashMap for a LinkedHashMap. This wouldn't be as easy if you were using Hashtable.
Since synchronization is not an issue for you, I'd recommend HashMap. If synchronization becomes an issue, you may also look at ConcurrentHashMap.
ConcurrentHashMap class is thread-safe i.e. multiple threads can operate on a single object without any complications.
Hashtable is considered legacy code. There's nothing about Hashtable that can't be done using HashMap or derivations of HashMap, so for new code, I don't see any justification for going back to Hashtable.
All the benefits of synchronization can be obtained more efficiently with Collections.synchronizedMap(HashMap).Neither Hashtable nor ConcurrentHashMap support null keys or values. HashMap does.
TreeMap is sorted according to the natural ordering of its keys, or by a Comparator provided at map creation time, depending on which constructor is used. This proves to be an efficient way of sorting and storing the key-value pairs.
